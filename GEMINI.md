# GEMINI.md — правила проекта CineHDR

Этот файл — обязательный контекст для ИИ-разработчика (Gemini) и людей-ревьюеров.
Каждое правило выведено из реальной ошибки, найденной при аудите кодовой базы.
Правила с пометкой **[MUST]** блокируют мерж. Правила **[SHOULD]** нарушаются
только с явным обоснованием в описании коммита.

---

## 1. Контекст проекта

CineHDR — форк видеоплеера [Cine](https://github.com/diegopvlk/Cine) (GTK4 / libadwaita /
libmpv / Python + ctypes) с поддержкой HDR-passthrough на Wayland.

**Стратегическая цель — upstreaming.** Каждое изменение оценивается по вопросу:
«сможет ли этот дифф стать частью PR в diegopvlk/Cine?» Форк — временная площадка,
а не самостоятельный продукт.

### Карта модулей и границы ответственности

| Модуль | Ответственность | Что ЗАПРЕЩЕНО делать в нём |
|---|---|---|
| `hdr_detection.py` | Единственное место детекции HDR-контента и возможностей системы | Хранить состояние; дублировать логику в других модулях |
| `hdr_controller.py` | Состояние HDR, GSettings, установка `target-*` свойств mpv | Работа с OpenGL; знание о виджетах |
| `gl_bindings.py` | ctypes-биндинги GL/EGL, константы | Логика уровня приложения |
| `gl_renderer.py` | RAII-жизненный цикл FBO/текстур | Знание об mpv и GTK-виджетах |
| `video_widget.py` | Интеграция mpv ↔ GTK scene graph | Бизнес-логика HDR (делегируется контроллеру) |
| `hdr_menu.py`, `hdr_diagnostics.py` | UI | Прямая запись свойств mpv (только через контроллер) |

Нарушение границ (например, `from .window import CineWindow` внутри callback) —
повод для отклонения на ревью.

## 2. Рендер-путь: инварианты [MUST]

Это самая опасная часть проекта. Ошибки здесь проявляются как
недетерминированные артефакты на конкретных драйверах и не ловятся тестами.

### 2.1. Контракт GdkGLTextureBuilder — священен

GTK может сэмплировать GL-текстуру **после** возврата из `do_snapshot`.
Поэтому:

- Текстура, обёрнутая в `builder.build()`, **не модифицируется**, пока GTK
  не сообщил об освобождении (destroy-notify) или не пройден GL fence.
- Используется ринг минимум из 2 текстур (рекомендуется 3) ЛИБО
  `builder.set_sync()` с `glFenceSync` после рендера mpv.
- Один и тот же texture id **никогда** не перерисовывается «на живую»
  между кадрами без синхронизации.

```python
# ЗАПРЕЩЕНО: рендер в ту же текстуру, что отдана GTK кадром раньше
self.mpv_ctx.render(opengl_fbo={"fbo": self.fbo_id.value, ...})
builder.set_id(self.texture_id.value)   # GTK ещё может читать её!

# ПРАВИЛЬНО: ринг буферов + fence
buf = self.ring.acquire_free()          # текстура, отпущенная GTK
self.mpv_ctx.render(opengl_fbo={"fbo": buf.fbo, ...})
fence = glFenceSync(GL_SYNC_GPU_COMMANDS_COMPLETE, 0)
builder.set_id(buf.texture)
builder.set_sync(fence)
texture = builder.build(lambda: self.ring.release(buf))
```

### 2.2. do_snapshot — только сборка сцены

Фаза snapshot должна быть дешёвой и без побочных эффектов для GPU:

- **ЗАПРЕЩЕНО** в `do_snapshot`: `mpv_ctx.render()`, `make_current()`,
  создание/ресайз FBO, любые блокирующие GL-вызовы.
- Рендер кадра mpv выполняется в tick-callback фрейм-клока
  (`add_tick_callback`) или в обработчике, инициированном `update_cb` mpv.
- `do_snapshot` лишь прикладывает последнюю **готовую** текстуру
  через `snapshot.append_texture()`.

### 2.3. Горячий путь — ноль аллокаций и ноль системных запросов

Всё, что выполняется на каждый кадр (snapshot, tick, property-доступ
из рендер-кода):

- **ЗАПРЕЩЕНЫ** вызовы `check_hdr_support()`, `Gdk.Display.get_default()`,
  `get_dmabuf_formats()` и любые capability-запросы. Результат кэшируется
  один раз и инвалидируется по сигналам (`notify::` смены дисплея/монитора,
  `changed::` GSettings).
- **ЗАПРЕЩЕНЫ** конструкции с жадно вычисляемым дефолтом:

```python
# ЗАПРЕЩЕНО: GLFramebufferResource() создаётся при КАЖДОМ обращении,
# даже когда атрибут существует
return getattr(self, "fbo_resource", GLFramebufferResource()).texture_id

# ПРАВИЛЬНО: атрибут гарантирован конструктором, обращение прямое
return self.fbo_resource.texture_id
```

- Импорты (`from gi.repository import Graphene`) — только на уровне модуля,
  не внутри функций горячего пути.

### 2.4. Жизненный цикл mpv render context

Порядок завершения фиксирован требованием libmpv:
`mpv_render_context_free()` → затем завершение ядра mpv.

- В `do_close_request` render context освобождается **до**
  `mpv.quit()` / `wait_for_shutdown()`, либо unrealize виджета гарантированно
  происходит раньше завершения ядра. Порядок должен быть явным в коде
  и подтверждён комментарием со ссылкой на доку libmpv.

## 3. Единственный источник истины [MUST]

Аудит нашёл две противоречащие функции детекции HDR (`utils.is_hdr_params`
считала HDR-ом сам факт primaries bt.2020) и два наблюдателя одного свойства.
Отсюда правила:

1. **Одна функция на одно понятие.** Детекция HDR-контента живёт только в
   `hdr_detection.is_hdr_content()`. Перед добавлением функции — `grep` по
   репозиторию: если похожая логика существует, расширяй её, а не дублируй.
2. **Один наблюдатель на одно свойство mpv на процесс.** Все подписки на
   `video-params` проходят через `HdrController`; UI получает изменения
   через его callback, а не через собственный `property_observer`.
3. **Один ключ настроек на одно понятие.** `hdr-mode` (`auto|force-hdr|force-sdr`)
   полностью описывает состояние. Дублирующий `hdr-enabled` запрещён —
   производные значения вычисляются, а не хранятся.
4. Детекция HDR требует **HDR-трансфер (PQ/HLG)**, а не только `sig-peak > 1.0`
   и не только primaries. Любое изменение критерия — с тестом на
   WCG-SDR-контент (bt.2020 + bt.1886), который НЕ должен триггерить HDR.

## 4. Правила для upstream-совместимости [MUST]

1. **Не переопределять инфраструктуру upstream.** Функции из `utils.py`,
   используемые кодом оригинала (`idle_add_once` и т.п.), не меняют семантику.
   Нужно другое поведение — добавь новую функцию с новым именем и используй
   её только в HDR-коде. (Реальный инцидент: `idle_add_once` был переписан
   в глобальный дедупликатор с локами — это молча изменило поведение всех
   вызовов оригинального кода.)
2. **Ребрендинг изолирован.** App id, имена gresource, конфиг-каталог,
   README, иконки — в отдельных коммитах с префиксом `brand:`, которые
   можно целиком исключить при подготовке PR в upstream. Функциональные
   коммиты не содержат ни одной строки ребрендинга.
3. **`po/` неприкосновенен.** Переводы Cine управляются через Weblate.
   Разрешено: добавлять файлы в `POTFILES.in`, писать английские msgid.
   Запрещено: править `.po`-файлы руками, перегенерировать их массово,
   переименовывать `.pot`.
4. **Не использовать устаревший `GSK_RENDERER=gl` для HDR [MUST]**. В GTK 4.14+
   и 4.16+ старый OpenGL-рендерер (`gl`) является устаревшим и не поддерживает
   цветовое управление (`Gdk.ColorState`) и 16-битные float-текстуры. Принудительная
   установка `os.environ["GSK_RENDERER"] = "gl"` (которая была в upstream SDR Cine)
   приводит к конвертации HDR-кадров в 8-битный sRGB и полному отключению HDR
   на Wayland. В CineHDR эта переопределение должно быть удалено, чтобы GTK
   использовал современные рендереры (`ngl` или `vulkan`).

## 5. Работа с API: не выдумывать и не лезть в приватное [MUST]

Правила специально для ИИ-разработчика — источник ошибок здесь
галлюцинированные или устаревшие API:

1. **Перед использованием свойства mpv — проверь его существование** в
   документации mpv (DOCS/man/options.rst, input.rst) для минимально
   поддерживаемой версии. Наблюдение несуществующих свойств
   (`display-hdr`, `icc-profile` как observable) запрещено, даже
   «безопасно» обёрнутое в try/except.
2. **Приватные API запрещены**: `mpv._get_property`, обход дерева виджетов
   GTK для доступа к внутренним `Gtk.Popover` внутри `GtkDropDown` и т.п.
   Если публичного API нет — фича не реализуется этим способом; ищи
   другой дизайн или открой issue.
3. Проверка возможностей — через версионные проверки/`hasattr` **в одном
   месте** (`hdr_detection`), с комментарием, какая версия GTK вводит API.
4. Если утверждение об API нельзя подтвердить ссылкой на документацию —
   считай его галлюцинацией и перепроверь, прежде чем писать код.

## 6. Обработка ошибок: падать громко в dev [MUST]

Найденный анти-паттерн: десятки `try/except Exception: print(...)`,
маскирующих реальные ошибки (именно так подписка на несуществующие
свойства прошла незамеченной).

- `except Exception` разрешён только на границах: callbacks из C-кода
  (GTK/mpv), завершение приложения, диагностический UI. В остальных
  местах перехватываются **конкретные** исключения.
- Каждый `except` либо восстанавливает состояние осмысленно, либо
  логирует через `logging` с уровнем и контекстом — `print()` в
  обработчиках ошибок запрещён.
- Guard-условия пишутся явно (`if not check_hdr_support(): return`),
  а не через «попробуем и проглотим».
- В dev-режиме (`run_dev.py`) допустимо включать режим, при котором
  «пограничные» перехваты перевыбрасывают исключение.

## 7. Definition of Done для изменений рендер-пути [MUST]

Изменение в `video_widget.py`, `gl_renderer.py`, `gl_bindings.py` или
`hdr_controller.py` считается готовым, только если:

1. Юнит-тесты `tests/test_hdr.py` проходят (`meson test`).
2. Прогнана ручная матрица (результат — таблицей в описании PR):

| Ось | Значения |
|---|---|
| Сессия | Wayland + HDR-дисплей, Wayland без HDR, X11 |
| GTK | ≥ 4.16, < 4.16 (fallback-путь) |
| GPU | AMD (Mesa), Intel (Mesa), NVIDIA (proprietary; GraphicsOffload отключён — HDR-путь проверять отдельно) |
| Контент | HDR10 (PQ), HLG, SDR, WCG-SDR (bt.2020 + bt.1886 — HDR НЕ должен включиться) |
| Сценарии | ресайз окна в проигрывании, смена монитора, fullscreen-переход, переключение force-hdr/force-sdr на лету, выход во время воспроизведения (нет краша при завершении) |

3. Нет новых аллокаций/системных вызовов в per-frame коде
   (проверяется чтением диффа, при сомнении — профилировкой).
4. Клетки матрицы, которые физически недоступны, помечаются
   `НЕ ПРОВЕРЕНО` — заявлять «работает» о непроверенном запрещено.

## 8. Дисциплина коммитов и PR [SHOULD]

1. **Один коммит — одна тема.** Запрещены коммиты вида
   «feat + refactor + CI + переводы + remediate audit findings»
   (реальный пример из истории: `7f5b319` смешал 4 несвязанных изменения).
2. Сообщения — conventional commits в стиле upstream (`fix:`, `feat:`,
   `refactor:`, `i18n:`, `brand:`). Без самоотчётов вида
   «remediate senior audit findings» — вместо этого описывай, **что**
   и **почему** изменилось; ссылки на пункты этого файла (`per GEMINI.md §2.1`)
   приветствуются.
3. **Документация синхронна с кодом.** Если коммит меняет поведение —
   в том же коммите обновляется README/metainfo. (Реальный инцидент:
   README заявлял флаг D-Bus `NON_UNIQUE` после того, как код вернул
   уникальный флаг.) Мёртвые аннотации и неиспользуемые импорты
   (`is_hdr_params` в `window.py`) удаляются в том же PR, где перестали
   быть нужны.
4. Целевой размер PR в upstream — до ~400 строк содержательного диффа.
   Больше — дели на серию.

## 9. UI и продуктовые ограничения [SHOULD]

1. Философия upstream — «Simple Design». Новые элементы UI по умолчанию
   минимальны: автоматика > настройки > диагностика. Любое расширение
   HDR-меню сверх текущего — только после согласования с мейнтейнером
   upstream (issue, не PR).
2. **Честность диагностики.** GTK не сообщает, реально ли композитор
   выводит HDR: он может тонмапить обратно в SDR. Диалог диагностики
   и документация обязаны формулировать это как «HDR-сигнал передан
   композитору», а не «HDR включён на экране».
3. Строки UI — только английские msgid, помеченные для перевода;
   формулировки — по GNOME HIG. Русский (и любой другой язык)
   попадает в приложение только через Weblate.

## 10. Чек-лист самопроверки перед каждым коммитом

Gemini обязан пройти список и указать в описании коммита пункты,
которые релевантны и выполнены:

- [ ] Нет дублирования существующей функции (grep выполнен)?
- [ ] Все использованные API mpv/GTK существуют в минимальных
      поддерживаемых версиях (проверено по докам, не по памяти)?
- [ ] В per-frame коде нет новых аллокаций, импортов, системных запросов?
- [ ] Текстуры, отданные GTK, не модифицируются без fence/destroy-notify?
- [ ] Ни одна функция upstream не изменила семантику?
- [ ] Нет правок в `po/`, кроме `POTFILES.in`?
- [ ] Нет `except Exception: print(...)` вне разрешённых границ?
- [ ] Нет обращений к приватным API (`_`-префикс, обход внутренностей виджетов)?
- [ ] README/metainfo соответствуют новому поведению?
- [ ] Коммит однотемный, ребрендинг не смешан с функциональностью?
- [ ] Утверждения «протестировано/работает» подкреплены реальным прогоном,
      непроверенное помечено как непроверенное?
